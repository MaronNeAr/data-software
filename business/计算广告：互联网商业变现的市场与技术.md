## 第一部分 计算广告基础

#### 名词解释解释

###### 广告商业化指标

- CPC：点击付费
- ADN：广告网络
- RPM：千次展示收益
- ADLOAD：广告加载率
- eCPM：千次展示期望收入
- CTR：广告点击率，广告点击与广告展现的比率
- CVR：转化率，下单于广告展现的比率

###### 广告平台

- TD：交易终端
- RTB：实时竞价平台（module）
- ADX：广告交易平台（module，广告引擎）
- SSP：提供方平台，媒体/流量主（module，用户技术中心后端）
- DSP：需求方平台，广告中/代理商（module，花火平台）

#### 计算广告基础

###### 广告有效性原理

- 曝光：广告位的物理属性，比如位置
- 关注：受众从物理上接触到广告到意识上注意到它的过程
- 理解：广告内容要在用户能理解的具体兴趣范围内；注意设定与关注程度相匹配的理解门槛
- 接受：广告的上下文环境对于广告的接受程度有着很大的影响
- 保持：希望 广告传达的信息给用户留下长久的记忆
- 决策：成功广告的最终作用是带来用户的转化行为

（越靠前的阶段对点击率的贡献越大，越靠后的阶段对转化率的贡献越大）

###### 互联网广告的技术特点

- 技术和计算导向：个性化广告推荐（实时特征和在线推理）和实时竞价
- 效果的可衡量性：实时计算点击率
- 创意和投放方式的标准化：根据人群包定向投放
- 媒体概念的多样化：整合营销在提高ROI的基础上并发掘潜在用户
- 数据驱动的投放决策：广泛收集用户的行为数据和广告反馈数据，利用云计算的基础设施对用户打上合适的标签，同样根据数据在多个广告竞争同一次展示时作出决策，再将投放的结果统计数据反馈给广告操作人员以调整投放策略

###### 计算广告的核心问题

- 核心优化问题：$max\sum^T_{i=1}\{r(a_i, u_i, c_i) - q(a_i, u_i, c_i)\}$
- a代表广告，u代表用户，c代表上下文，r代表收入，q代表成本

###### 广告收入的分解

- $eCPM = r(a, u, c) = \mu(a, u, c) \cdot \nu(a, u, c)$
- $\mu$代表点击率，$\nu$代表点击价值
- $eCPM$表示期望千次展示收益
- $RPM$表示千次展示收入
- $CPM$表示千次展示成本

###### 结算方式和eCPM估计的关系

- CPM结算：千次展示结算
- CPC结算：千次点击结算
- CPS结算：转化订单数结算
- CPA结算：转化行为数结算
- ROI结算：投入产出比结算
- CPT结算：独占时段结算

###### 受众定向方法

- **地域定向**：服务仅限特定区域进行定向
- **人口属性定向**：根据年龄、性别、教育程度、收入水平打标签
- **频道定向**：根据供给侧频道类型进行流量定向
- **上下文定向**：关键词、主题，也可以是根据广告主需求确定的分类
- **行为定向**：根据历史行为信息进行定向
- **精确位置定向**：利用蜂窝数据、GPS定位精确地址进行定向
- **重定向**：根据历史购买/订阅信息重定向
- **新客推荐定向**：用户选择感兴趣的内容标签，系统推荐定向
- **团购**：针对同一区域广告主，利用价格工具进行转换

###### 展示量合约

- **流量预测**
  - **售前指导**：提前约定曝光总数，并判断人群流量大小
  - **在线流量分配**：一次曝光满足多个合约（一个用户属于多个人群包），怎样决策整体满足所有合约
  - **出价指导**：广告主需要根据预计的出价了解一下可能获取到多少流量
- **流量塑形**：在某些活动期间向特定频道导流提高商业变现效率
- **在线分配**：系统通过“与”逻辑把流量切分成一个个独立的小格子，而不同的广告主（需求）会根据自己的条件，同时指向同一个格子，从而产生了**竞价**或**流量分配策略**的需求

#### 搜索与竞价广告

###### 搜索广告产品

- **变形能力**：eCPM远远高于一般的展示广告
- **受众定向标签**：基于上下文的搜索查询进行定向
- **展示形式**：需要与自然内容展示形式非常接近
- **产品策略**
  - **精确匹配**：将用户输入的查询分词，关键词集合与广告主设定的关键词一致即触发精确匹配
  - **短语匹配**：当用户的查询完全包含广告主关键词及关键词（包括关键词的同义词）的插入或颠倒形态时，就认为匹配成功
  - **广泛匹配**：当用户的查询词与广告主的关键词高度相关时，即使广告主并未提交这些查询词，也可能被匹配（数据挖掘）
  - **否定匹配**：明确指出哪 些词是不能被匹配的，这样可以灵活地关停一些低效的流量

###### 位置拍卖与机制设计

- **纳什均衡状态**：$\mu_t(\nu_s - q_s) \ge \mu_t(\nu_t - q_t)$
- **广义第二高价(GSP)**：在只有一个位置的拍卖中，向赢得该位置的广告主收取其下一位广告主的出价
- **VCG定价**：对于赢得了某个位置的广告主，其所付出的成本应该等于他占据这个位置给其他市场参与者带来的价值损害
- **市场保留价**：通过拟合其被保留价截断的分布为核心，可以通过理论上的计算公式来解出最优起价
- **广告网格**：竞价方式不向广告主做量的约定、淡化媒体和广告位
- **产品策略**：
  - **广告检索**：搜索广告检索与搜索基本一致，用常规的倒排索引技术、向量化检索就可以解决
  - **广告排序**：竞价广告中排序的准则是eCPM，而在CPC结算的情形下，对eCPM 的估计转化为对点击率的估计问题

###### 竞选广告需求放产品（DSP）

- **搜索引擎营销（SEM）**：搜索广告对应的需求方产品，即通过竞价采买搜索引擎关键词来做推广
  - **传统方式**：广告主根据到达率、自己网站的转化率和客单价来估计点击价值，并以此点击价值为参考来出价
  - **优化方式**：不同关键词上做不同的出价才能更深入地优化整体采买的ROI
- **媒体购买平台**：面向展示广告网络的一站式采买平台称为媒介采买平台
  - **技术挑战**：在ROI的优化部分，要合理地选择需要的受众定向条件，并在每个人群上优化出价，以保证量的要求和优化收益是一个复杂的组合优化问题
  - **技术方向**：广告网络中非 RTB 流量的ROI优化也成为广告技术产品

#### 程序化交易广告

<img src="/Users/bilibili/Library/Application Support/typora-user-images/image-20260131202921764.png" alt="image-20260131202921764" style="zoom:40%;" />

- **实时竞价（RTB）**：是一套**通信协议/流程**，规定了大家怎么出价、多少毫秒内必须返回结果
- **广告交易平台（ADX）**：核心枢纽连接买卖双方，发起竞价（RTB），并决定谁赢
- **需求方平台（DSP）**：代表广告主（比如某游戏公司），负责挑选流量、出价、管理预算和素材
  - **主要功能**：具备点击率预测、点击价值估计、流量预测、站外推荐等多方面的运算能力
- **供应方平台（SSP）**：代表流量主，负责管理广告位，决定什么时候、在哪里露广告，并把流量打包发出去
- **品牌广告**： 追求长期的认知，不求用户立刻下单，比如首屏广告：可口可乐宣传片
- **效果广告**： App下载、商品购买、表单提交，通常是 **CPA** (按行动计费) 或 **oCPX** (按点击扣费，但按转化优化)

###### 实时竞价（RTB）

- **核心瓶颈：Cookie Mapping（身份对齐）**

  - **痛点：** 浏览器有“同源策略”。ADX 域名下的 Cookie，DSP 是读不到的。比如 ADX 知道用户是 `ID_123`，但 DSP 的数据库里这个用户叫 `USER_ABC`

  - **Cookie 映射：** 在这笔请求发生前，ADX 和 DSP 已经通过某种同步机制（通常是像素埋点重定向）交换过 ID 了
  - **逻辑：** 请求传过来 `ADX_ID: 123`，DSP 查表：“哦，这就是我那里的 `USER_ABC`，他昨天在我这看了三款显卡，是个大金主！”

  - **后端挑战：** 这张映射表极大，且读取速度必须是微秒级的

- **毫秒级决策：DSP 的“灵魂三问”**：当 DSP 拿到 Cookie 并查出用户画像后，它要在 **20-40ms** 内完成以下逻辑：

  - **我认识他吗？**（画像匹配）

  - **他值多少钱？**（根据 pCTR/pCVR 算 eCPM）

  - **我要出价吗？**（检查余额、频率控制、投放策略） 如果算下来 eCPM 够高，DSP 就会返回一个带有 `Price` 和 `Creative_URL`（广告素材地址）的响应

- **时间片（Time Window）：ADX 的“倒计时”**
  - **硬限制：** 比如设定为 **50ms**
  
  - **胜出者逻辑：**在 50ms 内返回的 DSP 进入竞价池；超过 50ms 才返回的（由于网络抖动或 DSP 服务器慢），即便出价 100 块，ADX 也会直接抛弃（Timeout），视为弃权
  
  - **结果分发：** ADX 选出最高价（或二价逻辑下的赢家），把结果塞给媒体网站（SSP）
  
- **工程痛点**
- **高并发异步 I/O：** ADX 不能顺序等 DSP 回复，必须通过 **NIO (如 Netty) 或 Go 协程** 同时向几十个 DSP 发请求
  
- **定时器的精确性：** 那个“固定的时间片”需要非常精准，在 Java 里，你可能需要用到 `HashedWheelTimer`（时间轮算法）来处理海量的超时任务
  
- **内存压力：** 每一个询价请求在等待期间都会占用内存。如果 QPS 是 10w，等待时间 50ms，意味着内存里随时挂着 5000 个上下文对象

###### 广告交易平台（ADX）

- **ADX产品策略**：所有的广告竞价都是实时进行，因此不需要保存广告库，因而也不需要广告检索流程，排序过程也非常简单
- **结算方式**：ADX一般用CPM方式结算，广告网格一般用CPC结算
- **PMP交易平台**
  - **品牌安全（Brand Safety）：** 顶级品牌（如 LV、劳力士）不希望自己的广告出现在低质或违规的小 App 里
  - **流量质量：** 顶级媒体（如 B 站的开屏、核心信息流）不想在公开市场被“贱卖”，他们想把好位置留给优质客户

###### 需求方平台（DSP）

- **核心特征**：RTB方式的流量购买；需要支持需求方定制化的用户划分
- **产品策略**：检索、排序、定价、出价
- **出价策略**：eCPM 估计足够精准，并按照此值出价即可
  - **考虑因素**：周末/节假日、性别、品类
- **重定向**
  - **品牌广告**：用户已经选择过某品牌的服务或产品后，如果在比较高质量的媒体上看到该品牌的广告，他会进一步肯定自己决策的正确性，从而对该品 牌的认知度也大大加强
  - **效果广告**：当用户曾经考虑过某种产品，但没有完成最终转化，通过 在线广告将这个用户找回，点击率和转化率都会明显高于平均水平
  - **个性化重定向**：采用合适的创意推动他尽快完成转化行为；使 用推荐技术向其展示相关的商品以提升二次购买率
  - **搜索重定向**：将搜索过与广告主直接相关的关键 词的用户群作为重定向集合

###### 供应方平台（SSP）

- **产品逻辑**

  > 当广告请求到达时，首先检查优先销售的订单有无需求，这包括CPT和CPM的合约
  >
  > 如果有需求，按照优先级和在线分配的方案完成投放；如果没有这类销售合约，则进入竞价流程
  >
  > 竞价时，从自运营广告主库中找出eCPM较高的，并估算可供调用的若干 广告网络的eCPM，在这两者之间找到较高的广告候选，再以此作为MRP，通过RTB接 口向接入的各DSP实时询价

###### 数据加工与交易（ETL）

- **有价值的数据来源**
  - **用户标识**：用户Cookie、苹果IDFA、安卓IMEI
  - **用户行为**：转化（conversion）、预转化（preconversion）、搜索广告点击（sponsored search click）、展示广告点击（ad click）、搜索点击（search click）、搜索（search）、分享（share）、页面 浏览（page view）、广告浏览（ad view）
  - **人口属性**：性别、年龄、地域、分区、偏好内容
  - **地理位置**：用户当前所在国家、城市、街道
  - **社交关系**：关注、粉丝、朋友列表

###### 广告投放引擎

- **广告投放机**：接受广告前端Web服务器发来的请求，完成广告投放决策并返回最后页面片段的主逻辑（QPS和低延迟）
- **广告检索**：这部分的主要功能在线时根据用户标签与页面标签从广告索引（ad index）中查找符合条件的广告候选
- **广告排序**：这部分是在线高效地计算广告的 eCPM，并进行排序的模块
- **收益管理**：我们用这部分来统一代表在各种广告系统中将局部广告排序的结果进一步调整，以全局收益最优为目的做调整的功能， 如GD系统中的在线分配、DSP 中的出价策略等
- **广告请求接口**：根据前端接口形式的不同，广告请求可能来自于基于HTTP的 Web服务器，也可能来自于移动 App内的 SDK等
- **定制化用户划分**：由于广告是媒体替广告主完成用户接触，那么有时需要根据广告主的逻辑来划分用户群

###### 离线数据处理

- **用户会话日志生成**：各个渠道收集的日志需要先整理成以用户ID为键的统一存储格式，这样的日志称为用户会话日志
- **行为定向**：这部分功能完成挖掘用户日志，根据行为给用户打标，并将结果存储在用户标签的在线缓存中，供广告投放机使用
- **上下文定向**：这部分包括半在线页面抓取和上下文页面标签的缓存，与行为定向互相配合， 负责给上下文页面打上标签，用于在线的广告投放中基础知识准备
- **点击率建模**：它的功能是在分布式计算平台上训练得到点击率的模型参数和相应特征，加载到缓存中供 线上投放系统决策时使用
- **分配规划**：这部分为在线的收益管理模块提供服务，它根据广告系统全局优化的具体需求，利用离线日志数据进行规划，得到适合线上执行的分配方案
- **商业智能**：这部分包括 ETL 过程、dashboard 和 cube，这些是数据处理和分析流程的总括，担负着对外信息交流的任务
- **广告管理系统**：这部分是广告操作者，即客户执行（Account Execute， AE）与广告系统的接口

###### 在线数据处理

- **在线反作弊**：实时判断流量来源中是否有作弊流量，并且将这部分流量从后续的计价和统计中去除掉，是广告业务非常重要的部分
- **计费**：这部分同样是计算广告关键的业务功能之一。对于那 些经过扣费预算耗尽的广告，系统必须马上通知广告索引系统将其下线
- **在线行为反馈**：将短时内发生的用户行为和广告日志及时地加工成实时用户标签以及实时的点击率模型特征
- **实时索引**：部分的主要功能是实时接受广告投放数据，建立倒排索引

#### 基础知识准备

###### 信息检索

- **倒排索引**：从大量文档中查找包含某些词的文档集合这一任务用O（1）或O（log n）的时间复杂度完成
- **向量空间模型**：信息检索中最基础且最重要的文档相似度度量方法之 一
- **TF-IDF**：TF（Term Frequency）代表词频，也就是某文档中该词出现的次数，IDF（Inverse Document Frequency）代表倒数文档频率，即该词在所有文档中出现的频繁程度的倒数

###### 最优化方法

- **拉格朗日法与凸优化**：$L(\lambda, \nu) = \displaystyle\inf_x[f(x) + \lambda^Tg(x) + \nu^Th(x)]$
- **KKT条件**：是一组关于x，λ，ν的等式和不等式方程
- **下降单纯形法**：给定一个初始区间，假设区间内有唯一的最小值，可以按照黄金分割的方法不断缩小区间以得到最小值
- **黄金分割法**：$\frac{1-k}1 = \frac k{1-k}$
- **梯度下降法**：梯度的几何意义是 f 在 x点函数值上升最快的方向，因此它是一个与 x维数相等的矢量
- **拟牛顿法**： 不去算那个沉重的 $H$ 矩阵，而是用一个 **近似矩阵 $B$** 来代替
- **BFGS / L-BFGS：** 这就是拟牛顿法的代表
  - **L-BFGS（Limited-memory BFGS）：** 它不存储整个大矩阵，只存储最近 $m$ 次迭代的**位移差**和**梯度差**
  - **逻辑：** 就像你优化一个高并发系统，你不需要每次都去扫描全量数据库（全量 Hessian），你只需要根据最近几次的性能指标波动（梯度变化），反推系统的瓶颈在哪，然后快速调整
- **置信区间（Trust-Region）：** 像是一个资深的架构师
  - **策略：** 我先定义一个**“安全区”**（Radius $\Delta$），在这个圈子里，我觉得我对系统的理解（模型近似）是靠谱的
  - **执行：** 在这个圈子里找到最优解
  - **反馈：** 如果实际效果和预测一致（Good!），我就**扩大**安全区，走得更大胆点

###### 统计机器学习

- **最大熵（Max Entropy）：** 想象在写一个 `switch-case` 逻辑，如果对某些输入完全没把握，最稳妥的做法是让输出概率**均匀分布**
- **指数族分布（Exponential Family）：** 这就是一套**标准接口（Interface/Abstract Class）**，逻辑回归、高斯分布、泊松分布，全都是这个接口的实现类
- **混合模型（Mixture Model）：** 比如广告点击，有些人是手滑（分布 A），有些人是真有需求（分布 B）。你看到的是一堆混合后的日志，不知道每一条属于哪个分布
- **EM 算法（Expectation-Maximization）：** 这是一个标准的 `while` 迭代：
  1. **E-Step (Expectation)：** “猜”。基于现在的参数，给每条日志打个标签，猜它是“手滑”还是“需求”
  2. **M-Step (Maximization)：** “改”。根据刚才猜的结果，重新计算分布 A 和 B 的参数，让它们更契合这些数据
  3. **Loop：** 重复这个过程，直到参数不再变化
- **非贝叶斯（频率派）：** 点击率 = 0/0，程序崩溃（或直接设为 0），这太粗鲁（匪气）
- **贝叶斯派（文明人）：** 虽然还没数据，但我根据这个广告所属的行业（先验），给它一个初始值。随着展示次数（数据）增加，我的预估会越来越准

###### 广告检索

- **倒排索引的新问题**

  - 广告的定向条件组合可以看成是一个由与或关系连接的布尔表达式，这样的文档显然与搜索引擎面对的BoW文档不太一样，这里存在着有针对性的检索性能优化空间
  - 由上下文关键词或和用户标签比较丰富时，广告检索中的查询可能相当长，甚至会由上百个关键词组成，这种情况下的检索也与搜索引擎中主要由 1～ 4 个关键词组成的查询有很大区别

- **布尔表达式的检索**

  - **DNF**：每个 DNF 都可以分解成一个或多个交集（conjunction）的并，每个交集又可以进一步分解为一个或多个赋值集（assignment）的交

  - **Assignment/Conjunction/DNF**

    ``` c++
    class Assignment {
      public:
      	int attribute;	// 该Assignment指定的标签类型
      	bool belong;		// 表示该Assignment是属于还是不属于
      	int value;			// 该Assignment指定的标签取值
    };
    
    // 一个或多个Assignment的交
    typeof vector<Assignment> Conjunction;
    
    // 一个或多个Conjunction的并
    typeof vector<Conjunction> DNF;
    ```
